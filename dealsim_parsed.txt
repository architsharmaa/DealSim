Data  Model  
The  platform  stores  structured  information  related  to  organizations,  users,  training  simulations,  
roleplay
 
sessions,
 
and
 
evaluation
 
results.
 
Since
 
the
 
system
 
deals
 
heavily
 
with
 
conversation
 
transcripts,
 
analytics
 
metrics,
 
and
 
evolving
 
evaluation
 
structures
,
 
MongoDB
 
is
 
used
 
as
 
the
 
primary
 
datastore.
 
Why  MongoDB  
MongoDB  fits  this  use  case  because  simulation  sessions  contain  nested,  semi-structured  data  
such
 
as
 
transcripts,
 
analytics
 
snapshots,
 
conversation
 
state,
 
and
 
summaries.
 
Storing
 
these
 
artifacts
 
within
 
a
 
single
 
document
 
simplifies
 
retrieval,
 
avoids
 
join-heavy
 
queries,
 
and
 
allows
 
flexible
 
schema
 
evolution
 
for
 
rubrics
 
and
 
evaluations.
 
The
 
system
 
is
 
organized
 
into
 
the
 
following
 
MongoDB
 
collections.
 
1.  Organization  -  Represents  a  company  or  team  using  the  training  platform.  Fields:  ●  _id  ●  name  ●  industry  ●  createdAt  
Each  organization  can  have  multiple  users,  simulations,  and  training  sessions.  
2.  User  -  Represents  a  user  within  an  organization.  Fields:  ●  _id  ●  organizationId  ●  name  ●  email  ●  role  (organization_admin  |  org_employee)  ●  createdAt  ●  Roles:   ○  organization_admin  –  Creates  simulations  and  assigns  training.  
---PAGE---
 
○  org_employee  –  Participates  in  simulations  and  views  reports.  
 3.  Persona  -  Represents  the  simulated  buyer  persona  used  during  roleplay  sessions.  Fields:  ●  _id  ●  organizationId  ●  name  ●  role  ●  company  ●  personalityTraits  ●  resistanceLevel  ●  defaultObjections[]  ●  createdAt  
Personas  define  the  behavioral  characteristics  of  the  AI  buyer  during  the  simulation.  
4.  Context  -  Represents  the  sales  situation  used  in  a  simulation.  Fields:  ●  _id  ●  organizationId  ●  product  ●  dealSize  ●  salesStage  ●  specialConditions  ●  createdAt  
Context  provides  the  business  background  that  guides  AI  responses.  
5.  Rubric  -  Defines  how  simulation  performance  will  be  evaluated.  Fields:  ●  _id  ●  organizationId  ●  name  ●  competencies[]  ●  createdAt  ●  Each  competency  contains:  ●  name  ●  description  
---PAGE---
 
●  weight  ●  scoringGuidelines  
Rubrics  allow  organizations  to  evaluate  sessions  using  different  training  frameworks.  
6.  Simulation  -  A  simulation  combines  persona,  context,  and  rubric  to  create  a  reusable  
training
 
scenario.
 
When
 
a
 
simulation
 
is
 
created,
 
the
 
Prompt
 
Registry
 
generates
 
an
 
orchestrated
 
system
 
prompt
 
that
 
combines
 
persona
 
instructions,
 
context

and
 
evaluation

This

is

so

can

instantly
 
without

prompt

Fields:  ●  _id  ●  organizationId  ●  name  ●  personaId  ●  contextId  ●  rubricId  ●  orchestratedPrompt  ●  createdAt  Instead  of  a  single  string  prompt,  we  store  structured  prompt  components .  Example:  "orchestratedPrompt":  {   "systemPrompt":  "...",   "personaPrompt":  "...",   "contextPrompt":  "...",   "evaluationInstructions":  "..."  }  This  design  enables  reusable  components  to  be  combined  into  multiple  simulations.  
7.  Session  -  Represents  a  single  roleplay  interaction  between  a  user  and  the  AI  buyer.  
Session

MongoDB

Fields:  ●  _id  ●  simulationId  ●  userId  ●  status  (in_progress  |  completed  |  evaluated)  
---PAGE---
 
●  transcripts[]  ●  analyticsSnapshots[]  ●  conversationState  ●  summary  ●  overallSummary  ●  keyEvents[]  ●  evaluationId  ●  createdAt

Fields:  ●  wpm  ●  fillerWordCount  ●  talkRatio  ●  monologueFlag  ●  timestamp

Evaluation  results  are  displayed  in  the  simulation  report  dashboard.

1.  Organization  APIs  -  These  APIs  manage  organization-level  configuration.  Create  Organization  POST  /organizations  -  Creates  a  new  organization  account.  Get  Organization  GET  /organizations/{organizationId}  -  Returns  organization  details.  
2.  User  APIs  -  These  APIs  manage  users  within  an  organization.  Register  User  POST  /users  -  Creates  a  new  user  in  the  organization.  User  Login  
---PAGE---
 
POST  /auth/login  -  Authenticates  a  user  and  returns  a  session  token.  Get  User  Profile  GET  /users/{userId}  -  Returns  details  for  a  specific  user.  
3.  Persona  APIs  -  These  APIs  allow  administrators  to  manage  reusable  buyer  personas.  Create  Persona  POST  /personas  -  Creates  a  new  buyer  persona  configuration.  List  Personas  GET  /personas  -  Returns  all  personas  for  the  organization.

5.  Rubric  APIs  -  These  APIs  define  evaluation  frameworks  used  in  simulations.  Create  Rubric  POST  /rubrics  -  Creates  a  rubric  containing  evaluation  competencies.  List  Rubrics  GET  /rubrics  -  Returns  available  rubrics.

training
 
scenarios.
 Create  Simulation  POST  /simulations  Creates  a  new  simulation  and  generates  the  orchestrated  system  prompt. End  to  End  Session  Flow  
This  section  describes  how  a  simulation  session  is  executed  from  initialization  to  evaluation.  
1.  Session  Initialization  
A  user  selects  a  simulation  and  starts  a  session.  
POST  /sessions  
The  Simulation  Service  retrieves  the  simulation  configuration  and  its  orchestrated  prompt  
from
 
MongoDB.
 
A
 
new
 
session
 
document
 
is
 
created
 
with:
 
status:  in_progress  transcripts:  []  analyticsSnapshots:  []  conversationState:  {}  
The  session  ID  is  returned  to  the  frontend  and  a  WebSocket  connection  is  established  for  
real-time
 
analytics.
 
2.  User  Sends  Message  
When  the  user  sends  a  message:  
POST  /sessions/{sessionId}/message  
The  message  is  appended  to  the  session’s  transcripts  array .  
Example:  
---PAGE---
 
{  speaker:  "seller",  content:  "...",  wordCount:  12,  timestamp:  "..."  }  
3.  Conversation  Engine  Processing  
The  AI  Roleplay  Engine  constructs  the  LLM  prompt  using:  
systemPrompt  personaPrompt  contextPrompt  conversationState  recentTranscripts  
The  request  is  forwarded  to  the  LLM  Gateway ,  which  generates  the  AI  buyer  response.  
The  response  is  appended  to  the  session  transcript.  
4.  State  &  Analytics  Update  
The  Conversation  State  Engine  updates  structured  state:  
claimsMade  topicsCovered  objectionsRaised  objectionsResolved  
The  Analytics  Engine  computes  metrics:  
wpm  fillerWordCount  talkRatio  monologueFlag  
Analytics  snapshots  are  stored  and  pushed  to  the  frontend  via  WebSocket.  
5.  Session  Completion  
When  the  user  ends  the  session:  
---PAGE---
 
POST  /sessions/{sessionId}/end  
The  session  status  is  updated  to  completed ,  and  the  Evaluation  Pipeline  is  triggered.  
6.  Evaluation  &  Summary  
The  evaluation  engine  processes:  
session  transcripts  simulation  rubric  conversation  summary  structured  evaluation  prompt  
It  generates:  ●  competency  scores  
 ●  transcript  evidence  
 ●  feedback  
 ●  overall  score  
 
The  system  also  generates  a  session  summary  and  key  events ,  which  are  stored  with  the  
session.
 
7.  Results  Display  
The  frontend  retrieves  results:  
GET  /sessions/{sessionId}/evaluation  
The  simulation  report  displays  evaluation  scores,  feedback,  transcripts,  and  session  analytics.
